## 1. Executive Summary

DMart (Avenue Supermarts Ltd.) operates one of India's largest retail chains with 350+ stores across 15+ states. The current reporting infrastructure relies on manual Excel-based monthly reports generated by store managers and regional heads, creating significant delays in decision-making, data inconsistencies across regions, and inability to perform real-time or cross-functional analysis.

This Business Requirements Document defines the scope, objectives, stakeholders, functional requirements, and success criteria for the DMart Sales Analytics Dashboard — a Power BI-based reporting solution built on the retail_rolap star schema database.

The solution will replace manual reporting with an automated, interactive dashboard enabling revenue tracking, customer segmentation, store performance benchmarking, product category analysis, and inventory monitoring — all accessible to business stakeholders in real time.

---

## 2. Business Context & Problem Statement

### 2.1 Current State (AS-IS)

| Area | Current Situation | Business Impact |
|------|-------------------|-----------------|
| Reporting | Manual Excel reports, updated monthly | 15-20 day lag in decision-making |
| Data Access | Store managers email CSV files to HQ | No central visibility across regions |
| Customer Insight | No segmentation or RFM analysis done | Unable to identify at-risk or high-value customers |
| Product Analysis | Category performance tracked via pivot tables | No margin vs revenue cross-analysis |
| Inventory | Stock reports generated weekly per store | Out-of-stock situations not flagged proactively |

### 2.2 Desired State (TO-BE)

- Live Power BI dashboard connected directly to retail_rolap MySQL database
- Role-based access — Store Manager, Category Head, Regional Director, CFO views
- Automated daily refresh — no manual data entry required
- Cross-functional analysis — revenue, margin, customer, inventory in one platform
- Mobile-accessible reports for field managers

### 2.3 Business Problem Statement

DMart's business leaders lack timely, accurate, and integrated visibility into sales performance, customer behavior, and store operations. Manual reporting processes create data silos, prevent proactive decision-making, and consume 40+ hours of analyst time per month.

Without a centralized analytics solution, DMart risks delayed response to revenue drops, inability to identify top/bottom performing stores, and missed opportunities for customer retention and cross-selling.

## 5. Project Scope

### 5.1 In Scope

- Power BI dashboard with 5 report pages: KPI Dashboard, Store Performance, Product & Category, Customer Analysis, Trends
- Direct MySQL connection to retail_rolap database (fact_sales, dim_*, agg_* tables)
- DAX measures for all KPIs: Total Revenue, Profit, Margin %, Avg Basket Size, MoM Growth, Return Rate
- Customer RFM segmentation using fact_sales and dim_date
- Scheduled daily data refresh via Power BI Gateway
- Row-level security (RLS) — Store Managers see only their store data
- Mobile layout optimization for field managers

### 5.2 Out of Scope

- Real-time streaming data (dashboard refreshes daily, not live)
- Predictive analytics or machine learning models
- Integration with external competitor data
- Modifications to the retail_rolap database schema
- Online channel data (currently not in retail_oltp)

## 8. Data Requirements

### 8.1 Source Database

- **Database:** retail_rolap (MySQL 8.0 — Star Schema for Analytical Processing)
- **Host:** localhost / production MySQL server (IT to confirm production host)
- **Refresh Method:** Power BI Gateway — Scheduled daily refresh at 06:00 AM IST

### 8.2 Tables Required

| Table Name | Type | Usage in Dashboard |
|------------|------|-------------------|
| fact_sales | Fact Table | Core revenue, profit, margin, discount calculations — all KPI measures |
| dim_date | Dimension | Time intelligence — year, month, quarter, weekday for all trend charts |
| dim_product | Dimension | Category, brand, margin%, perishable flag for product analysis |
| dim_store | Dimension | Store name, city, state, store_type, city_tier for store analysis and map |
| dim_customer | Dimension | Gender, age_group, customer_segment, loyalty_member for customer analysis |
| dim_supplier | Dimension | Supplier type, payment_days, preferred vendor flag for supplier page |
| agg_monthly_category_sales | Aggregate | Monthly category trend chart on Trends page |
| agg_monthly_store_performance | Aggregate | Monthly store trend for top 5 stores |
| agg_customer_lifetime | Aggregate | CLV segment analysis — Platinum, Gold, Silver, Bronze |

## 10. Assumptions, Constraints & Risks

### 10.1 Assumptions

- retail_rolap database is fully populated with at least 2 years of transaction data before dashboard go-live
- Power BI Pro or Premium licenses are available for all target stakeholders
- IT team will configure Power BI On-Premises Gateway for MySQL connectivity
- dim_customer[customer_segment] is already computed in ETL (VIP, Premium, Regular, Occasional, Inactive)
- agg_customer_lifetime table is refreshed daily as part of the ETL pipeline

### 10.2 Constraints

- Budget: No additional data warehouse tooling — solution must use existing retail_rolap MySQL schema
- Timeline: Dashboard must be delivered within 4 weeks of requirements sign-off
- Technology: Power BI Desktop for development, Power BI Service for publishing
- Data: No real-time streaming — daily refresh only

### 10.3 Risk Register

| Risk | Probability | Impact | Mitigation |
|------|-------------|--------|------------|
| MySQL Gateway connectivity issues in production | MEDIUM | HIGH | IT to test gateway 1 week before go-live. Fallback: CSV export from MySQL |
| Data quality issues in retail_rolap (nulls, wrong joins) | LOW | HIGH | UAT testing with Finance team to validate all KPI values against source |
| Low stakeholder adoption of Power BI | MEDIUM | MEDIUM | Training sessions for each stakeholder group. Simple navigation design |
| Scope creep — additional visuals requested post sign-off | HIGH | LOW | Change request process defined. All new requirements go through BA review |
